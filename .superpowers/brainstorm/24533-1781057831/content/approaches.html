A

Incremental — Componente por Componente

Estratégia: Refinar seção por seção, começando pelo nav e descendo até o footer

Prós

  • Progresso visível rapidamente
  • Fácil revisar cada seção individualmente
  • Pode parar/ajustar no meio se necessário
  • Commits atômicos e organizados

Contras

  • Site fica inconsistente durante implementação
  • Mais commits (bom para git, mas mais verbose)
  • Precisa testar integração no final

Ordem sugerida:

  1. Criar SVGs das variações de logo
  2. Nav (logo + estrutura)
  3. Hero (tipografia + tagline)
  4. Seção Sobre
  5. Preocupações + Especialidades + FAQ
  6. Footer
  7. Ajustes globais CSS (tipografia, espaçamentos)
B

Foundation First — Base Depois Componentes (recomendado)

Estratégia: Primeiro criar todos os SVGs e ajustar CSS global (tipografia, espaçamentos, cores), depois aplicar nos componentes

Prós

  • Estabelece sistema consistente desde o início
  • Componentes herdam boas práticas automaticamente
  • Menos retrabalho — acerta a base primeiro
  • Alinhado com design system thinking
  • Mais rápido no total

Contras

  • Progresso visual mais lento no início
  • Precisa planejamento mais cuidadoso

Ordem sugerida:

  1. Criar todos os SVGs (logo-nav-desktop, logo-nav-mobile, logo-footer)
  2. Atualizar variáveis CSS (tamanhos tipográficos, espaçamentos)
  3. Ajustar classes base (h1, h2, h3, section-label, etc)
  4. Aplicar logos no nav/footer
  5. Refinar cada seção seguindo as bases
  6. Polimento final
C

Big Bang — Tudo de Uma Vez

Estratégia: Criar branch, implementar tudo, testar, e fazer um único merge/commit

Prós

  • Consistência garantida — tudo muda junto
  • Um único commit limpo
  • Fácil reverter se necessário

Contras

  • Sem progresso visível até o final
  • Difícil revisar mudanças grandes
  • Se algo der errado, mais difícil debugar
  • Menos oportunidade para feedback incremental

Execução:

  1. Criar branch feature/brand-refinement
  2. Implementar todas as mudanças
  3. Testar tudo localmente
  4. Merge para main com um commit
  5. Deploy

💡 Recomendação

Abordagem B: Foundation First é a mais adequada para este projeto. Estabelece bases sólidas (SVGs, CSS system) primeiro, depois aplica nos componentes. Resulta em código mais limpo, menos retrabalho, e mantém commits organizados sem perder visibilidade do progresso.
